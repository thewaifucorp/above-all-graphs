`refactor.rs` implements the two graph-driven refactor tools described in `SPEC.md` section 7: coordinated multi-file rename, and `affected`, a dependency-aware "what tests might this break" query for CI or pre-commit hooks. Impact analysis for browsing blast radius before editing lives separately in `crate::impact`; this module is the one place in `aag` that actually writes to a user's source files.

## Rename

`Command::Rename` in `main.rs` wires to `rename_run(root, old_name, new_name, write)`. Because rename mutates real files, `rename_run` previews by default and only applies changes when `write` is `true`. The planning logic is split out into `rename_plan`, which stays pure (no filesystem writes) so both the CLI preview and any future MCP surface can show the same result. `rename_plan` opens the existing graph, finds the node matching `old_name`, and fails with `Error::SymbolNotFound` if there's no match or `Error::AmbiguousRename` if more than one symbol shares the name — rename refuses to guess.

Once a unique target is found, `rename_plan` collects every file that would need editing: the target's own declaration file, plus every caller file reached via `Graph::callers`, walked through the same confidence-tagged edges the rest of the graph uses. It deliberately skips two edge classes: any edge whose `Confidence` is `Ambiguous`, and any edge of `EdgeKind::Explains`. The `Explains` exclusion matters because it means rename never touches documentation prose that merely mentions a name (that's `crate::docs::describe`'s domain, not this module's) — renaming an identifier inside docs is a different, riskier operation. Each candidate file's actual occurrence count is computed with `whole_word_count`, a scan that only counts matches at identifier word boundaries (via `is_word_boundary_match` / `is_ident_char`), so `Foo` doesn't touch `FooBar`. Results come back as a sorted `Vec<RenameChange>`, each pairing a `file_path` with its `occurrences` count. `format_plan` renders this into the human-readable dry-run text printed when `write` is `false`.

When `write` is `true`, `rename_run` calls `rename_apply`, which re-reads each planned file, runs `whole_word_replace` (the same boundary-aware logic as the count, but substituting `new_name`), writes the file back, then reindexes the whole repo via `crate::resolve::index_repo` — the same full-rebuild path the file watcher uses, so the graph reflects the new name immediately after rename returns. `rename_plan` currently ignores its `new_name` parameter for anything beyond that reindex; it's reserved for a future dry-run diff preview that would show exact before/after lines rather than just counts.

## Affected

`Command::Affected` reads a list of changed file paths from stdin (typically piped from `git diff --name-only`) and calls `affected_run(root, stdin)`, which trims and filters blank lines then delegates to `affected(root, changed_files)`. `affected` seeds a BFS frontier with every graph node declared in one of the changed files, then walks callers outward — again via `Graph::callers` — up to `MAX_DEPTH` hops (20, mirroring `crate::impact::MAX_DEPTH`'s rationale) to find every file transitively depending on something that changed. The visited file set is then filtered twice: `looks_like_test_file` keeps only paths that look like tests (matching patterns like a `tests/` directory component, `test_`/`_test.` naming, or `.test.`/`.spec.` suffixes), and the changed files themselves are excluded from the result — the tool's whole value is surfacing *downstream* impact, not restating what's already known to have changed.

Both `rename_plan` and `affected` return `Error::IndexMissing` if `root` has no index yet, since both require `Graph::open_existing` to succeed before doing any graph walking. Like the rest of `aag`'s cross-file resolution, both tools are whole-word, name-based heuristics rather than full semantic analysis — fast and good enough for the confidence-tagged graph model the rest of the codebase relies on.
