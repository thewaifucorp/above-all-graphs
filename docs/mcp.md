`mcp.rs` implements the same JSON-RPC handler over two transports. `run` serves newline-delimited stdio. `run_http` serves `POST /mcp` on `127.0.0.1`, returns 202 for notifications, rejects non-local browser origins, and optionally requires `Authorization: Bearer <token>` through `--api-key` or `AAG_MCP_API_KEY`. `GET /mcp` returns 405 because AAG currently uses request/response JSON rather than server-sent events. Both transports reconcile once and start the same incremental watcher.

Dispatch happens in `handle`, which switches on the JSON-RPC `method` field: `initialize` returns protocol version and server info, `ping` returns an empty result, `tools/list` returns the tool schemas via `listed_tools`, and `tools/call` goes through `call_tool`. Unknown methods produce a JSON-RPC error object unless the request was a notification, in which case `handle` returns `None` and nothing is written.

Tools are declared centrally in the `TOOL_SPECS` array of `ToolSpec` structs, each carrying a name, description, single primary argument name/description, and an `implemented` flag. This is the source of truth CLAUDE.md refers to: `explore` is the only tool in `DEFAULT_LISTED_TOOLS`, so `tools/list` only advertises `explore` by default. The rest of `TOOL_SPECS` — `node`, `search`, `callers`, `callees`, `impact`, `rename`, `affected`, `cypher`, `detect_changes`, `wiki`, `describe_doc` — become visible in `tools/list` only when their name appears in the comma-separated `AAG_MCP_TOOLS` environment variable, parsed by `enabled_tool_names`. Critically, this gating only affects what shows up in the tool list; `call_tool` looks up tools by name against the full `TOOL_SPECS` table regardless of `AAG_MCP_TOOLS`, so any tool is still callable via `tools/call` even when unlisted. This matches the rationale in the module doc comment: agents pick the wrong tool more often when offered many similarly named options, so most stay hidden from the menu but remain reachable.

Of the listed tools, `cypher` and `detect_changes` have `implemented: false` — calling them returns an `isError: true` response with a "not implemented yet" message rather than dispatching, via the check in `call_tool`.

Most tools take one string argument, whose name/description come straight from the `ToolSpec`; `tool_schema` builds the JSON schema for each, with `rename` and `describe_doc` special-cased because they take two required arguments (`name`/`new_name` and `doc`/`description` respectively) instead of one. Tool execution for the single-argument tools funnels through `dispatch`, which matches the tool name to the corresponding handler: `explore::format`, `explore::format_node` (for `node`), `search_text`, `edges_text` with `Direction::Callers` or `Direction::Callees`, `impact::format`, `write_wiki`, and `affected_text`. `write_wiki` always writes to `.aag/wiki` under the indexed root regardless of the `out_dir` argument, which is ignored. `rename` and `describe_doc` bypass `dispatch` entirely and are handled by `call_rename` and `call_describe_doc`, since their two-argument shape doesn't fit the single-`arg` `ToolSpec` model; `call_rename` drives `crate::refactor::rename_plan` and `crate::refactor::rename_apply`, applying the rename to disk and reindexing, and `call_describe_doc` calls `docs::format` to record a description linked to symbols it mentions.

Every tool response, implemented or not, is wrapped in the same shape: a `content` array with one `{"type": "text", "text": ...}` entry and an `isError` boolean, matching the MCP tool-result convention. Domain errors (missing symbols, storage failures) are caught and turned into `isError: true` text rather than propagating as JSON-RPC protocol errors — protocol-level errors are reserved for malformed requests, like a missing tool name or missing params.
